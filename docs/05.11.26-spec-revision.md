# triage-cli-revision-spec

CNC/site map and Datadog as the original spine. extract.py (189 LOC), datadog.py (187 LOC), apex-cnc-inventory.md, scripts/build_cnc_map.py, and a sizable share of the LLM site-extraction path serve a Datadog-window report that your described workflow does not ask for. The team has already demoted this in writing; the code has not caught up.
Inbox TUI (inbox/app.py 514 LOC plus widgets and tcss). It is a watcher viewer, not an investigation workspace. Your pipeline is single-ticket and interactive; the inbox does not fit.
Watch mode (watcher.py 282 LOC). Polling a Zendesk view to triage on a cron is orthogonal to your "I provide a ticket ID" entry point. Useful adjacent feature; not your spine.
Two LLM-call helpers tied to Datadog (extract_anchor, extract_site). Both exist to feed the Datadog window. With Datadog demoted, extract_site is dead weight and extract_anchor becomes optional.
Provider lock-in. Your prompt says "claude/codex," meaning the workflow should be model-agnostic. The repo enforces Claude Agent SDK only and writes that constraint into CLAUDE.md. That decision is correct for their authentication situation (no API key, enterprise OAuth seat), not for yours.
triage vs investigate as separate subcommands. The team's own review says triage should eventually call into the investigation service. Today there are two parallel report shapes; users have to know which to invoke.

What I would change
Order is by leverage, not size.

Add a Memory layer. A MEMORY.md plus a tiny SQLite index (sqlite-fts is fine; no embeddings, no vector DB). After every investigation, append three rows: customer + ticket subject + symptom, the assessment summary, and the final resolution if known. Before the LLM call, retrieve top-K similar prior entries via BM25/FTS against the new ticket's subject + first 500 chars, and pass them in the user message as "prior investigations" context. Two side benefits: duplicate detection becomes trivial, and the file is human-readable so the operator can prune or correct it. The team forbade embeddings explicitly; sqlite-fts respects that constraint.
Promote investigate to the spine; collapse the others. Make triage and watch subcommands call investigate internally with flags that control interactivity. Inbox stays as an opt-in viewer over saved reports. This is the slice the product-direction-review already prescribed; ship it.
Add Atlassian read-only enrichment, opt-in via flag. --with-jira and --with-confluence should hit the customer's REP/active issue search and the runbooks space. Treat them like Datadog: optional evidence sources behind an enrichment interface, not pipeline gates. The current "no Jira, no Confluence" rule was set to prevent scope creep; for your DailyNOC workflow it is a wrong default.
Add Zendesk customer-history pull. Three lines of additional code in zendesk.py to call /api/v2/search.json?query=requester:{email} type:ticket sort:updated_at. Limit to the last N tickets; fold into InvestigationEvidence as a new evidence type alongside LocalFileEvidence. This is your "preliminary investigation based on customer ticket history" step.
Provider abstraction behind an env switch. One thin interface with two implementations: Claude Agent SDK (current) and a Codex/OpenAI HTTP path. Gate by LLM_PROVIDER. Yes, the team's CLAUDE.md says do not do this. Your prompt says you want it. These two facts are incompatible; pick one. If you keep Claude-only, drop "codex" from your stated pipeline. If you want codex, you eat the provider layer.
Build the investigation-progress TUI. Spec exists at docs/superpowers/specs/2026-05-08-investigation-progress-tui.md, deferred until "operators complain." With a memory layer, Atlassian enrichment, and customer history, the agent has 5 to 7 sequential calls per ticket; the linear stderr stream will feel like a hang. Ship the three-pane progress view at the same time you ship the memory layer.
Ship the doctor subcommand from the Go-spike lessons doc. One screen of green/red checks for env vars, OAuth session, Atlassian creds, output dir writability. Cheap win, removes the worst onboarding pain.
Rename triage-cli to something the workflow earns. The product is no longer one-shot triage; it is guided investigation with cross-system context. The current name encodes the original drift.

Net
The repo is a well-engineered tool aimed slightly past your workflow. The interactive evidence-drop loop, workspace layout, redactor, and structured-output discipline are direct hits. The Datadog-and-site-map spine, the inbox, and the watcher are adjacent features wearing the spine's clothes. The two pieces that distinguish your described pipeline from "Claude Code with a Zendesk tool" are the cross-system context pull (Jira/Confluence/customer history) and the durable memory; neither exists. Build those two and the score moves from 62 to the high 80s without touching anything else.
